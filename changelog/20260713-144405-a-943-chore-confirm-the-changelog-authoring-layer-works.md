## Changed

- Verified the inherited changelog authoring layer works in the content-only,
  non-npm repo ([A-943](https://linear.app/acme-skunkworks/issue/A-943)): `pnpm validate:changelog` passes; the `changelog` skill
  writes valid dated entries; the `changelog-completeness` gate enforces the
  `feat`/`fix`/breaking PR-title ↔ entry coupling (exercised live in CI on the
  [A-942](https://linear.app/acme-skunkworks/issue/A-942) / [A-944](https://linear.app/acme-skunkworks/issue/A-944) feature PRs); and `changelog-enrich.yml` enriched a merged entry
  end-to-end.
- Aligned `changelog/README.md` with the deploy-target model: reframed it from
  "single, semver'd npm package" to "versioned, non-npm deploy target" (no
  npm/GitHub-package publish; the release-orchestrator cuts the git tag + GitHub
  Release), and replaced the stale `pkg-release.yml` / `mode: finalise`
  lifecycle with the in-repo `changelog-enrich.yml` (`mode: enrich`) leg. Also
  corrected the frontmatter-schema comments that described the post-merge fields
  as "filled at release (finalisation)" — they are filled post-merge by
  `changelog-enrich`.
- Noted that the per-skill `config.json` files are gitignored template-seed,
  regenerated by `initialise-skills` — so any residual `dist`/`files`/
  `publishConfig` advisory values there are out of the committed template surface
  and get correct deploy-target values when a spawned repo runs the init skill
  ([A-946](https://linear.app/acme-skunkworks/issue/A-946)).
